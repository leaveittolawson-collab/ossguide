Historical Context

40 Years of Open Source Licensing

How we got from a frustrated MIT lab to 90% of modern software.

1983
GNU Project Founded
Richard Stallman launches GNU Project after frustration with proprietary printer firmware. Defines the "four freedoms" of software.
1989
GPL v1 Released
The first General Public License introduces copyleft — the radical idea that freedom should be enforced through licensing.
1991
MIT License & Linux
Linus Torvalds releases Linux under GPL v2. MIT License gains traction at universities for its simplicity.
1998
OSI Founded, "Open Source" Coined
Eric Raymond and others create the Open Source Initiative, publishing the Open Source Definition with 10 criteria.
2004
Apache 2.0 — Patent Era Begins
Apache License 2.0 released with explicit patent grants — a critical addition as software patent wars intensify.
2007
GPL v3 & AGPL
GPLv3 tackles hardware locks (Tivoization) and DRM. AGPL closes the "SaaS loophole" — using OSS over a network without sharing source.
2018–2023
Commons Clause & Business Source
HashiCorp, MongoDB, Redis shift to Business Source License (BUSL) and Server Side Public License (SSPL) to prevent cloud giants from commoditizing their work.
2024–Now
AI Training Data Rights
New licensing questions emerge around whether using code to train AI models constitutes copyright infringement — a debate that will shape the next decade.

License FAQ

Without a license, copyright law gives you all rights by default — which means nobody else has the right to use, modify, or distribute your code. All rights reserved is the legal default. This is often unintentional and cuts off collaboration entirely.

Yes, but only if you own all the copyright (or have contributor agreements). If others have contributed under the original license, you'd need their permission to re-license. This is why companies like Mozilla have contributor license agreements (CLAs).

It depends on how you use it. Static linking typically triggers GPL copyleft requirements. Dynamic linking via LGPL is designed to avoid this. Network access (APIs) generally does not trigger GPL — though AGPL closes that loophole. This is a nuanced legal question where context matters enormously.

Functionally identical for most practical purposes. ISC is slightly shorter (no separate permission notice paragraph). ISC was created to eliminate ambiguity in the old BSD advertising clause. Node.js uses ISC for this reason. Either is a safe choice for permissive licensing.

AGPL requires that if you run AGPL software as a network service (SaaS), you must make the source code available to users of that service — even if you never distribute the software itself. This is why companies like Google have internal policies banning AGPL dependencies.

Generally no. Creative Commons licenses were designed for creative works (text, images, music) and lack provisions critical for software (warranty disclaimers, patent grants, source code requirements). For software, use OSI-approved licenses. Use CC for documentation and datasets.
